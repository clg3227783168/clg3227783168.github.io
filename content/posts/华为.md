# 总览
解决高并发场景下，因粗粒度读写锁导致的内存性能瓶颈问题
1.在高内存负载的情况下，系统卡顿非常严重。
2.通过perf、火焰图等手段发现内存回收上占用了大量CPU的时间片
3.根本原因是：保护页表操作的锁是粗粒度的读写锁，持有该锁时会锁住一片连续的具有相同读写执行属性的内存页.当触发pagefault时，需要建立虚拟页和物理页的映射关系；而想要回收某个物理页，要解除这个物理页和虚拟页的映射关系，都需要持有这个粗粒度的页表锁。
举个例子，不同CPU操作一个virtual region内的不同页面也会产生竞争。这可能导致可能仅仅是需要回收4KB的内存，就要一直等到整个10KB内存页都处于未被操作的时候才能进行，严重降低内存回收的效率。
因为往往在内存压力大的时候会触发大量内存回收的操作，所以持锁的时间要越短越好。
总结来说，页表锁必须要满足两个条件：
    1锁的粒度要细化，锁锁住的内存页要小
    2持锁的时间要越短,也就是锁的临界区要小。
所以我采用细粒度自旋锁的方案。并且所有可能发生睡眠或阻塞的操作（比如说页加载，文件类型的页加载可能会触发磁盘IO，需要等待IO返回）禁止在临界区内，确保新锁的临界区非常短，也保证了持锁的时间非常短
由于自旋锁只会忙等待，不会向读写锁一样在写锁争用时会阻塞线程，所以可以避免线程切换的上下文开销

难点：
并发场景下数据一致性的保证
假设我们需要修改某个页表项的值
在原有大锁下，保证查询到的页表项值可用
但是，由于耗时操作被移出临界区，现在这个页表项可能在查询值后，被使用前的这个时间间隙被其他线程修改
所以设计了一个解决方案：在提交最终修改之前，增加一次一致性检查，检查pte的值是否与查询到的值相等

工程适配的难点
由于逻辑比较复杂，涉及到的代码量比较大，所以采用渐进式的页表锁适配策略，先保留旧锁的状态下引入新锁，当一整条路径适配完毕后再将旧锁剥离

由于忙等待会占用CPU，所以在想能不能进一步减少持锁时间，开发了一套基于透明内核对象的快速系统调用接口。
将VMA，也就是描述进程地址空间的信息，包括起始/结束地址，读写执行权限，映射的文件列表等本来属于内核态才能看到的信息以只读的方式映射到用户态，让用户态能够直接看到这些信息，避免陷入内核查询信息的开销

# 读写锁允许多个读家进入，为什么在你的场景下性能反而不如自旋锁？
1. 因为系统的高内存压力体现在并发写竞争，pagefalut和内存回收都是大量写，这种情况下，读写锁需要维护等待队列和原子计数器，开销大，但是自旋锁只需要一个占bit，开销小。
2. 自旋锁在操作不同物理页的时候访问的是不同的锁变量，减少了多核之间同步的开销；读写锁的粒度更大，需要同步的概率更高。

# 难点: 如何处理多个线程同时为同一个虚拟地址分配了不同的物理页这种竞态情况
这是将 I/O 和物理页分配移出锁外后必然面对的场景。
解决逻辑：采用一致性检查，第一个持锁的成功，否则失败。
假设线程 A 和线程 B 同时检测到PTE为空，也就是说虚拟页没有物理页，触发缺页，它们都在锁外各自调用 alloc_page() 分配了不同的物理页 PageA和 PageB。
线程 A 先拿锁：进行一致性检查，发现PTE仍为空，于是将PTE指向PageA，释放锁，成功返回。线程B后拿锁，此时进行一致性检查，发现 PTE 已经不再是空（被 A 填入），必须负责释放掉自己刚刚申请的PageB。

悲观方案：
先加锁，再查询pte值，再执行分配物理页的操作，再修改pte值，再解锁。在此期间，不允许其他读写。
乐观方案：
先读取变量值，再分配物理页，再加锁锁住pte，再进行一致性检查，再解锁。在此期间，不允许其他读写。
所以乐观锁锁住的时间更短
而且乐观锁消除不必要的串行
在悲观方案中，物理页分配是串行的；在乐观方案中，它是并行的。
场景
10个 CPU 核同时触发缺页异常。悲观方案： 10个核必须排队。核1分配完内存，核2才能开始分配，以此类推。乐观方案： 10个核可以同时去申请内存。它们只在最后一步写入 PTE 时才需要排队。
这种方案看似浪费了一次内存分配的 CPU 周期，但在高并发场景下，“并发分配 + 失败清理” （乐观方案）的吞吐量远高于 “排队分配”（悲观方案），因为分配内存的动作（寻找空闲链表、还可能触发内存回收等）比释放内存要慢得多。由于锁的粒度很细，同一个虚拟页面的并发缺页概率很低，失败重试的概率小。


乐观方案的吞吐量更大，尾部延迟（多次pagefault，延迟时间最高的那几次延迟）更小，CPU 利用率更高->(有效计算/(有效计算+锁竞争+上下文切换))

# 进程挂起 阻塞 睡眠
阻塞：进程需要资源时触发，占用内存不占用CPU
挂起：系统内存不足时或者用户主动（Ctrl+Z）触发，获得资源时唤醒几乎不占用内存，只保留进程控制块PCB（内含寄存器状态），代码段（可能丢弃，因为磁盘里有一模一样的）、数据段和堆栈，不占用CPU。现代操作系统的做法是将进程的内存数据压缩，仍然放到内存中。
睡眠：进程主动sleep(n)停下来，系统的定时器队列定期检查唤醒睡眠时间到期的队列，
